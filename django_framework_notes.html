    Django Form API (django.forms.Form):
  • Purpose: Used for creating forms independent of database models.
  • Field Definitions: Form fields are defined manually using CharField, EmailField, etc.
  • Usage: Ideal for creating custom forms that don't directly map to database models or include fields from multiple models.
  • Validation: Provides form validation methods like is_valid().
  • Data Handling: Requires manual data processing and saving.
  • Integration with Models: Not directly tied to Django models.

    • Code forms.py

    from django import forms
    from django.core.validators import MinLengthValidator, EmailValidator
    class ContactForm(forms.Form):
       name = forms.CharField(max_length=100, validators=[MinLengthValidator(3)])
       email = forms.EmailField(validators=[EmailValidator()])
       message = forms.CharField(widget=forms.Textarea)

    Code views.py

    from django.shortcuts import render, redirect
    from .forms import ContactForm
    from .models import Contact # Import your Contact model
    def contact(request):
      if request.method == 'POST':
       form = ContactForm(request.POST)
        if form.is_valid():
          # Process valid form data
          name = form.cleaned_data['name']
          email = form.cleaned_data['email']
          message = form.cleaned_data['message']
          # Save data to Contact model
          contact_instance = Contact(name=name, email=email, message=message)
          contact_instance.save()
          return redirect('success_url') # Redirect to a success page
        else:
          form = ContactForm()
        return render(request, 'contact.html', {'form': form})
       

    ModelForm (django.forms.ModelForm):
  • Purpose: Designed for creating forms directly tied to Django models.
  • Field Definitions: Form fields are automatically generated based on the model's fields.
  • Usage: Suitable for forms related to CRUD operations on specific database models.
  • Validation: Includes model-level validation automatically.
  • Data Handling: Automatically handles data saving to the corresponding model instance.
  • Integration with Models: Tightly integrated with Django models, allowing direct interaction with model instances.

    • Code forms.py

    from django import forms
    from .models import Contact
    class ContactModelForm(forms.ModelForm):
       class Meta:
         model = Contact
         fields = ['name', 'email', 'age']
         # we can add custom things as well as per our need
          labels = {
         'name': _('Your Name'),
          'email': _('Your Email'),
          'age': _('Your Age'),
          }
          validators = {
          'name': [MinLengthValidator(3)],
          'email': [EmailValidator()],
          }

    Middleware
    Middleware

    Middleware are put in the settings.py file and execute in order in which they are defined

  • Request Middleware:
    1. Middleware classes defined in MIDDLEWARE setting are executed in the order they're listed.
    2. Each middleware class has access to the incoming HTTP request before it's passed to the view.
    3. You can perform operations like authentication, session management, or modifying request headers here.
  • View Function/Class:
    1. The processed request is passed to the appropriate view function or class-based view.
    2. Your view performs its logic and returns an HTTP response.
  • Response Middleware:
    1. Middleware classes are executed in reverse order after the view function or class.
    2. Each middleware class has access to the outgoing HTTP response.
    3. You can perform operations like setting response headers or modifying the response content here.

    • Middleware

    class based middleware Code:

    class MyMiddleware:
       def __init__(self, get_response):
         self.get_response = get_response
       def __call__(self, request):
         # Code to be executed for each request before the view is called
         # For example, you can log request data or perform authentication here
           response = self.get_response(request)
           # Code to be executed for each response before it's returned
           # For example, you can modify response headers or content here
       return response


    function based middleware Code:

    def MyMiddleware(get_response):
       def middleware( request):
         # Code to be executed for each request before the view is called

           response = get_response(request)
           # Code to be executed for each response before it's returned
           # For example, you can modify response headers or content here
          return response
       return middleware

    Authentication

    In Django REST Framework (DRF), authentication refers to the process of verifying the identity of users making requests to your API endpoints. DRF provides several built-in authentication classes that you can use to authenticate users. These authentication classes are pluggable, meaning you can choose which one(s) to use based on your application's requirements.
    REST_FRAMEWORK = {
       'DEFAULT_AUTHENTICATION_CLASSES': [
         'rest_framework.authentication.TokenAuthentication',
         'rest_framework.authentication.SessionAuthentication',
       ],
    }

    Steps involved in token based verification

    Obtaining a Token:

  • User Authentication:  The user sends their credentials (e.g., username and password) to the authentication endpoint of your API.
  • Authentication Process:   Django Rest Framework processes the request, authenticates the user based on the provided credentials.   If the credentials are valid, Django Rest Framework generates a token for the authenticated user.
  • Token Generation:   The token is typically a long random string associated with the authenticated user. This token is then returned in the response to the authentication request.

    • Using the Token:

  • Client Storage:   The client (e.g., a web browser, mobile app) stores the token securely. In web applications, tokens are often stored in cookies or local storage.
  • Subsequent Requests:   For any subsequent requests that require authentication, the client includes the token in the request headers, typically in the Authorization header.

    • Validating the Token:

  • Token Extraction:  When a request reaches a view protected by token authentication, Django Rest Framework extracts the token from the request headers.
  • Token Verification:   The extracted token is then verified for its validity. This typically involves checking if the token exists in the database of valid tokens and if it hasn't expired.
  • User Association:   If the token is valid, Django Rest Framework associates the token with a user. This information is then available in the request.user attribute.

            class IsAdminOrReadOnly(BasePermission):
            """
            Custom permission to allow only admin users to edit objects.
            """
            def has_permission(self, request, view):
                # Read permissions are allowed to any request
                if request.method in ['GET', 'HEAD', 'OPTIONS']:
                    return True
            
                # Write permissions are only allowed to admin users
                return request.user and request.user.is_staff
            
            class IsManagerOrReadOnly(BasePermission):
            """
            Custom permission to allow only manager users to edit objects.
            """
            def has_permission(self, request, view):
                # Read permissions are allowed to any request
                if request.method in ['GET', 'HEAD', 'OPTIONS']:
                    return True
            
                # Write permissions are only allowed to manager users
                return request.user and request.user.groups.filter(name='manager').exists()
